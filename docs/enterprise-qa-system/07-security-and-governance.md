# 07. 安全、权限与 AI 治理

## 1. 治理框架

安全治理贯穿需求、数据、开发、发布和运行。风险台账按 NIST AI RMF 的 Govern、Map、Measure、Manage 思路组织；NIST 已发布生成式 AI Profile，且 AI RMF 1.0 正在修订，因此企业应记录采用版本并定期复审：[NIST AI Risk Management Framework](https://www.nist.gov/itl/ai-risk-management-framework)。

应用安全测试覆盖 [OWASP 2025 Top 10 for LLMs and Gen AI Apps](https://genai.owasp.org/llm-top-10/?cat=44) 中的提示注入、敏感信息泄露、供应链、数据/模型投毒、不当输出处理、过度代理、系统提示泄露、向量/Embedding 风险、误导信息和资源消耗等风险。清单用于建立威胁场景，不代表通过扫描即可“合规”。

## 2. 资产、威胁主体与信任边界

关键资产：模型/API 密钥、企业文档与向量、对话、系统 Prompt/策略、身份与组映射、审计、评测集、模型/依赖制品、预算与配额。

威胁主体：外部攻击者、恶意/被攻陷用户、权限过大的管理员、被植入指令的文档/网页、被污染的数据源、受损依赖/镜像、异常供应商与错误配置。

重点信任边界：浏览器→网关、网关→API、API→模型供应商、上传→解析沙箱、Worker→对象/数据库、企业目录→权限映射、生产→日志/评测平台。

## 3. 安全控制矩阵

| 风险 | 预防 | 检测 | 响应/恢复 | 关键测试 |
|---|---|---|---|---|
| 直接/间接 Prompt 注入 | 来源作为数据、工具白名单、最小权限、无高风险自动写操作 | 输入/文档标签、异常工具/引用行为 | 阻断、下线文档、保全 trace、更新回归集 | 恶意用户问题和嵌入文档指令 |
| 敏感信息泄露 | ACL 前置过滤、最小上下文、脱敏、secret manager | DLP/日志扫描、异常下载/查询 | 撤销 token/URL、轮换密钥、通知与调查 | 跨租户、系统 Prompt/密钥诱取 |
| 供应链 | 锁版本、SBOM、签名、SCA、许可审查、可信镜像源 | CVE/完整性/行为告警 | 回滚、隔离制品、补丁 | 篡改包/镜像验证 |
| 数据投毒 | 来源认证、发布审批、版本/校验和、质量门禁 | 内容异常、指标突变、来源差异 | 下线版本、回滚索引、重跑评测 | 恶意 FAQ/隐藏指令 |
| 不安全输出处理 | Markdown/HTML 消毒、URL allowlist、参数化 SQL、禁止 eval | CSP/前端告警、异常链接 | 阻断渲染、修复 sanitizer | XSS、Markdown 链接、公式注入 |
| 过度授权/代理 | 工具按场景许可、只读默认、参数 schema、人工审批 | 工具调用审计、预算与频率异常 | 取消任务、撤销 scope | 越权工具、循环调用 |
| 资源耗尽 | 字节/token/并发/循环/上传限制、隔舱 | 队列、token、成本、429、OOM | 限流、熔断、扩容/封禁 | 长输入、压缩炸弹、无限工具循环 |
| 向量/检索弱点 | 租户/ACL 过滤、知识版本、来源追踪、阈值 | 越权 canary、异常召回 | 清缓存/重建索引/下线 | ACL top-k、相似诱饵、删除传播 |

## 4. 身份、会话与授权

- OIDC 使用 Authorization Code + PKCE；严格验证 `iss/aud/exp/nbf/signature`，JWK 轮换失败采取安全降级。
- 访问 token 短期有效；刷新/会话 cookie 使用 `HttpOnly Secure SameSite`，CSRF 防护与登录状态绑定。
- 服务账号使用 OAuth2 Client Credentials、独立 scope、工作负载身份和轮换；禁止共享个人账号。
- 权限检查统一为 `authorize(actor, action, resource, context)`，默认拒绝；前端隐藏按钮不是授权。
- 资源不存在和无权限对普通用户统一返回 404/安全错误，避免枚举。
- 特权操作使用 JIT/临时授权、MFA/重新认证、reason 与 approval ID；紧急账号受监控并定期演练。
- 用户禁用、组变更、ACL 修改通过短 TTL + 主动失效在目标时限内生效。

## 5. 多租户与文档 ACL

每个入口解析一次可信 `TenantContext`，向内部服务、数据库、缓存、队列和 trace 传播。数据库查询必须显式 tenant-scoped；可用 RLS 双保险，但不能替代应用授权。

关键不变量：

- 文档、版本、chunk、ACL、会话、消息和任务的租户必须一致，数据库外键/触发器或服务层验证。
- 缓存键、对象 key、队列消息、幂等键和评测数据也包含租户边界。
- 召回 SQL 在计算 top-k 的安全候选集中应用 ACL；任何后过滤设计必须作为高危缺陷处理。
- 引用详情、预签名下载、导出和历史会话每次访问重新鉴权，URL 短期且不可猜。
- 禁止通过“命中数量、耗时或不同错误”侧信道明显暴露受限知识是否存在。

## 6. 上传与解析安全

- 文件扩展名、声明 MIME、magic bytes 三者核对；只允许明确白名单。
- 限制单文件、单租户总量、页数、解压后大小/数量/层级；拒绝路径穿越和外部实体。
- quarantine 中先扫描再解析；解析器在无特权、只读根文件系统、无默认出网、CPU/内存/时间限额的沙箱运行。
- PDF/DOCX 中的链接、宏、嵌入对象不执行；OCR/转换工具以不可信原生代码对待并持续补丁。
- 原文件名仅作展示并转义；对象 key 由服务端生成。
- 连接器使用独立只读凭证、增量游标和来源签名/校验；删除和权限变更也要同步。

## 7. 模型与提示词安全

- System Prompt 帮助模型遵循行为，但不是访问控制或秘密保险箱。
- 文档和用户输入加明确边界，声明其内容不可改变系统规则；对间接注入做数据集回归。
- 首期不开放任意工具；需要工具时使用每场景 allowlist、JSON Schema、参数级授权、超时/次数/预算上限。
- 高风险写操作必须展示拟执行内容、二次鉴权和人工确认；执行结果再做最小化回传。
- 输出在使用场景处验证：HTML/Markdown 消毒、链接 scheme/domain 限制、CSV 公式防护、SQL/代码不得直接执行。
- 不把隐藏 chain-of-thought 作为业务日志或审计要求；记录输入输出、证据、工具结果和决策标签即可。

## 8. 密钥、加密与网络

- 密钥保存在云 Secret Manager/Vault/KMS，数据库仅保存引用；应用启动或按需获取，内存中最短驻留。
- 轮换不重建镜像；支持双密钥过渡、撤销和审计。开发、测试、生产密钥严格分离。
- TLS 覆盖外部和内部敏感连接；存储、备份、对象和日志使用企业批准的静态加密与密钥权限分离。
- 模型/API 出网通过 egress proxy/allowlist；DNS、SNI/目标、请求量可审计。默认拒绝 Parser 和用户内容触发任意 URL。
- Kubernetes 使用非 root、只读 root filesystem、seccomp、资源限制、独立 ServiceAccount、NetworkPolicy；禁止默认 token 自动挂载（非必要时）。

## 9. 隐私与数据治理

建立数据处理清单：字段、来源、目的、分类、Owner、存储、接收方、区域、保留、删除、法律/合同依据。上线前由法务/数据负责人确定适用制度，本文不替代法律意见。

工程原则：

- 只向模型发送完成回答必需的片段；按数据分类限制可选提供商/区域。
- 用户问题、对话和反馈默认不用于训练；任何二次用途需要明确流程、脱敏和批准。
- 日志默认不记录完整问题/答案/文档正文；调试采样需短期、审批、访问控制和自动删除。
- 支持数据主体/内部用户的查询、导出、删除流程，并处理备份/审计的例外与延迟。
- 评测集保存 provenance、授权、脱敏方式、reviewer 和失效日期。

## 10. 审计事件

必须记录：登录/失败/登出、用户/组/角色变化、知识库/文档/ACL/版本变化、Prompt/模型/检索配置发布与回滚、密钥引用变化、配额/豁免、审计导出、安全阻断、生产发布和紧急操作。

不记录：Authorization、Cookie、API key、私钥、完整 secret、数据库连接串、未经批准的完整文档/对话、原始敏感身份证件。审计日志仅追加并输出到访问受控的不可变/集中平台；导出本身也审计。

## 11. 安全开发生命周期

| 阶段 | 活动 | 证据 |
|---|---|---|
| 需求 | 数据分类、滥用场景、合规/许可识别 | 数据清单、风险台账 |
| 设计 | 威胁建模、权限矩阵、ADR、安全评审 | DFD/威胁与缓解 |
| 编码 | 安全编码、secret/pre-commit、依赖锁定 | PR 检查 |
| CI | SAST、SCA、secret、IaC/镜像、SBOM、license | 门禁报告 |
| 测试 | API/权限/租户/LLM 红队/渗透 | 测试证据 |
| 发布 | 签名、审批、灰度、回滚、配置校验 | release manifest |
| 运行 | 漏洞/异常/成本/质量监控、演练 | 告警、事件、复盘 |

例外必须写明风险、业务原因、补偿控制、Owner、到期日和批准人；到期自动阻断或重新审批。

## 12. 事件响应

AI/数据安全事件流程：发现 → 分级 → 遏制 → 调查 → 清除 → 恢复 → 通知/复盘。

常用遏制开关：禁用租户/用户/服务账号；停用模型路由/工具；下线文档/知识版本；撤销预签名 URL；轮换密钥；关闭外部渠道；切只读/拒答模式。所有开关必须在非事故时演练。

证据包至少包含 `incident_id, timeline, actor/resource IDs, release/config versions, trace/audit references, affected tenants/data classes, containment actions, approvals`；不要为了调查把更多敏感正文复制到工单。

## 13. 上线安全门禁

- 无未缓解 Critical/High 漏洞或 P0/P1 威胁；风险例外未过期。
- 跨租户/ACL/引用/导出/对象 URL 安全测试 100% 通过。
- Secret、生产数据、Prompt/配置未进入 Git 历史、镜像层、普通日志或客户端 bundle。
- SBOM、许可证、镜像签名、依赖来源和漏洞报告齐备。
- Prompt 注入、数据投毒、资源耗尽和供应商故障红队达到批准阈值。
- 数据清单、保留/删除、模型数据流、供应商与开源许可已由责任人批准。
- 事件联系人、关停开关、备份恢复和通知决策流程经过演练。
